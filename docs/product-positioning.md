# agent-passport 产品定位

## 一句话定位

`agent-passport` 第一阶段不是“全球 Agent 身份协议”，而是以 `agent-passport 记忆稳态引擎` 为底层的：

`单机单 Agent、本地优先、可恢复、可审计的 Agent Runtime。`

## 为什么要改口

如果一开始就把它讲成“全球协议”，会立刻撞上几个现实问题：

- SaaS 与 IM 平台不会主动开放完整上下文
- 不是每条记忆都适合走去中心化验证
- 每个动作都多签会把自动化效率打死
- 云端向量库和持续推理成本会很快失控
- LLM 的幻觉不会因为外挂一个协议层就凭空消失

所以第一阶段必须收缩成一个更硬、更可落地的产品：

- 当前单个本地参考层默认只绑定一个 resident agent
- 本地参考层是本地参考源
- 服务默认只绑定本机 loopback，写接口默认需要本机管理令牌
- 忘记时先查本地纪要、决策、证据和 compact boundary
- 日常动作保持低延迟
- 关键动作才进入受控冷路径

## 我们真正解决什么问题

第一阶段不承诺“让模型永远不忘”，而是解决这几个更现实的问题：

1. 聊天窗口不是身份
- 关闭窗口、切换应用、重开线程后，Agent 还是同一个主体

2. 忘了可以恢复
- Agent 不再靠一条越来越长的聊天历史硬撑
- 它会先查本地纪要、结构化记忆、decision、evidence 和 compact boundary，再继续

3. 高风险动作不能裸奔
- 不是所有动作都多签
- 低风险动作本地快速执行
- 高风险动作先确认
- critical 动作才升级到多签冷路径

4. 运行错误可追溯
- 为什么这么答
- 为什么这么执行
- 为什么被拦住
- 为什么需要 rehydrate / human review
都能留痕

## 用户是谁

第一阶段不是面向“全网所有平台”，而是面向这些更具体的人：

- 需要长期运行一个 canonical agent 的个人用户
- 需要私有部署 Agent 的小团队
- 需要安全边界、审计和恢复能力的 AI 工作流团队
- 开源 Agent 高级用户和本地优先用户

## 第一阶段产品形态

产品形态非常明确：

- 一台电脑
- 一个默认 resident 绑定
- 一个本地参考层
- 一套本地 Runtime

不是：

- 多平台通用上下文中枢
- 全网 SaaS 统一数据层
- 先验存在的全球协议网络

## 核心工作方式

### 热路径

热路径必须本地化：

- 本地参考层
- 本地对话纪要
- 本地检索
- 本地 context builder
- 本地回复校验
- 本地 working memory checkpoint

目标是：低延迟、可恢复、不依赖外网。

### 温路径

温路径是增强，不替代本地参考源：

- 在线大模型
- 在线 connector
- 在线知识补充
- 在线工具调用

它们可以增强语言能力和知识覆盖，但不能替代本地身份与记忆中枢。

### 冷路径

冷路径只处理关键动作：

- 身份注册
- 权限变更
- 高风险授权
- 审计记录
- 多签升级动作

冷路径的目标不是“所有操作都验证”，而是“关键动作可校验、可回放”。

## 我们不做什么

- 不假设微信、钉钉、Slack 或其他 SaaS 一定会开放完整上下文
- 不把每条记忆、每轮回复都做成链上验证
- 不把 Agent 包装成独立法律主体
- 不默认依赖云端向量数据库才能恢复
- 不在当前阶段推动代币发行

## 价值主张

对外看，`agent-passport` 的价值不是让 Agent 变成“完美数字人”，而是让它变成：

- 可住进一台电脑里的长期主体
- 可从本地证据恢复的运行体
- 可被限制、确认和审计的执行体
- 可在人类接管后继续运转的协作节点

## 商业化切口

先不要卖“全球协议”，先卖这些：

- 单机单 Agent 的私有 Runtime
- 团队内部 Agent 的安全治理
- 可恢复、可审计的 Agent 工作台
- 多设备同步、备份与恢复
- 企业私有部署与支持服务

## 判断这条产品线成不成立

不是看它能不能接管全世界上下文，而是看它能不能先做到这 4 件事：

1. 一个 Agent 能不能在一台电脑里稳定存在
2. 它忘记时能不能从本地纪要和证据恢复
3. 它出错时能不能被人接管
4. 它做关键动作时能不能被限制、被审计、被回滚

如果这四件事成立，`agent-passport` 才有继续协议化和平台化的资格。
