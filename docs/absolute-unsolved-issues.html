维护状态：持续更新的长期文档
当前版本日期：2026-04-03
适用范围：只记录“不是靠我们继续写代码就能从根本上彻底解决”的问题。
文档规则
1. 这份文档只收录“绝对解决不了”或“不是本项目单方面能从根本解决”的问题。
2. 纯工程问题、架构问题、实现问题，不放进本清单；那些问题必须继续被彻底解决。
3. 后续对话中一旦再次确认新的“根本不可解项”，统一追加到这份文档并重新导出 Word。

一、当前确认的绝对不可彻底解决项

编号 问题 为什么不能由这套引擎单独彻底解决 我们现在能做到的最强替代方案
1 Agent 本身成为独立法律责任主体 这是法律、司法解释、监管与法域制度问题，不是协议层或 Runtime 自己能决定的。
现阶段 Agent 不是天然法律主体，产品不能单方面“赋予”它法律责任能力。 		把责任链说清楚：
委托人是谁、运营主体是谁、谁授权、谁确认、谁执行、谁回滚、谁审计。
2 强制让封闭平台开放底层上下文 / 记忆 / 会话 API 微信、钉钉、Slack、OpenAI、Anthropic 等平台是否开放入口，取决于它们自己的商业策略、监管要求和生态控制权。
本项目无法单方面迫使这些平台交出底层上下文。 		不依赖全网开放。
只接有权限的 connector、官方 bot、用户明确授权的导出、webhook、文件与本地事件。
3 彻底消灭 LLM 幻觉、上下文坍缩与非确定性 当前大模型本身具有概率性、混沌性和上下文退化问题。
外挂一个协议层、记忆层或回复校验器，无法从物理和算法层“绝对消灭”这些现象。 		只能做“损害约束”：
本地参考源、分层记忆、最小上下文重建、回复校验器、checkpoint、rehydrate、人工接管。
4 绝对零风险安全 任何真实系统都不能诚实承诺“绝对零风险”。
本地机器、操作系统、依赖链、供应链、人员误操作、社会工程、未来漏洞，都可能突破当前安全边界。 		只能尽量缩小爆炸半径：
loopback、鉴权、加密、Keychain、恢复包、capability allowlist、子进程隔离、审计、回滚。
5 协议 adoption / 生态网络效应本身 “全球 Agent 协议”是否真的被广泛采用，不是靠代码质量单方面决定的。
它依赖 Killer App、市场时机、生态合作、标准竞争和分发渠道。 		先把产品收缩成可独立成立的 MVP：
单机单 Agent、本地优先、可恢复、可审计的 Runtime。

二、不应放进本清单的问题

以下问题不是绝对不可解，所以不能被拿来“逃避继续修复”：

  • API 鉴权不够细
  • 本地存储加密不够强
  • Keychain 迁移链不完整
  • sandbox 还不够强
  • 风险分级策略还不够精细
  • 本地检索召回还不够强
  • 读权限还不够细粒度
  • 恢复链、审计链、回滚链还不完整

这些都属于“工程上必须继续彻底解决”的范围。

三、当前执行原则

  • 能彻底解决的工程问题，继续解决，不得借口“太难”跳过。
  • 不能彻底解决的世界性问题，必须诚实记录，不能对内对外假装已经解决。
  • 产品叙事必须基于真实边界，不得把“缓解”包装成“消灭”。
