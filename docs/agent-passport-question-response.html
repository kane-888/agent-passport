Agent Passport 问题回应、解决方案与落地检查

版本：2026-04-03
项目：Agent Passport
当前阶段定位：单机单 Agent、本地优先、可恢复、可审计的 Agent Runtime

一、执行摘要

结论：Agent Passport 不能按“全球去中心化 Agent 协议”直接推进，但可以按 “单机单 Agent、本地优先、风险分级、关键动作可校验”的 Runtime 成立。

用户提出的问题集中在四个层面：平台封闭、效率损失、成本失控、以及 LLM 本身不可控。 这些问题都是真问题，不能靠更大的叙事掩盖。

因此，当前解法已经收缩为一个更现实的版本：

  • 不再假设全网平台会开放完整上下文
  • 不再把多签用作所有动作的默认机制
  • 不再把向量库当作第一阶段的基础前提
  • 不再声称 Passport Store 可以“驯服”LLM，只把它定义成本地参考源

二、用户提出的关键问题

  1. “去中心化”与“实时性”的冲突：链上验证、分布式查询、多签会拖慢 Agent。
  2. 责任主体模糊：Agent 违约或侵权时，责任归谁。
  3. 记忆主权难落地：RAG 仍有噪声，结构化记忆如何避免幻觉。
  4. 大厂生态封闭：微信、钉钉、Slack、SaaS 不会主动开放底层上下文。
  5. 先鸡还是先蛋：没有 Killer App 的协议底座容易自嗨。
  6. 隐私与安全风险：Passport 汇聚身份、记忆、权限、资产，单点风险很高。
  7. 多签授权与自动化的悖论：每个动作都多签会让 AI 失去效率。
  8. 算力与存储成本失控：四层记忆系统如果全量云化，会非常烧钱。
  9. “本地参考层”过度乐观：LLM 即使读到正确凭证，仍然会幻觉。

三、现在确定的产品解法

1. 产品定位重写

新定位不是“全球 Agent 身份协议”，而是： 当前 Passport store 默认只绑定一个 resident agent 的本地 Runtime。

  • 聊天窗口只是入口，Passport Store 才是本地参考源
  • 忘了先查本地纪要、decision、evidence、compact boundary
  • 低风险动作保持低延迟，关键动作再进入受控冷路径

2. 热路径 / 温路径 / 冷路径分层

路径 目标 做什么 为什么这样设计
热路径 低延迟 本地 Passport Store、本地检索、Context Builder、Verifier、Checkpoint 避免把每轮回复都拖进链上验证或复杂远程依赖
温路径 增强能力 联网大模型、知识包、可选 connector 增强语言能力，但不替代本地参考层
冷路径 可校验 DID、VC、关键授权记录、多签升级动作 只用于低频高风险动作，避免实时性被拖垮

3. 多签改成风险分级，而不是默认官僚化

多签不是默认执行系统，而是 critical 动作的保险丝。

风险等级 默认策略 适用动作 设计目标
Low Discuss / Auto Execute 搜索、本地纪要记录、读取、总结、恢复 保持低延迟，不让日常动作失速
Medium Discuss 计划更新、普通外部动作准备、策略讨论 先协商，不急着执行
High Confirm 修改重要资料、影响外部结果的执行动作 要有明确确认
Critical Multisig 删除、撤销、转移、轮换密钥、改 signer、不可逆动作 只把真正危险的动作升级到冷路径

4. 本地检索优先、非向量优先

第一阶段默认不依赖云端 Vector DB，也不把向量检索当作恢复的前提。

  • 默认策略：local_first_non_vector
  • 默认 scorer：lexical_v1
  • 默认：allowVectorIndex=false
  • 优先源：structured memory、conversation minutes、decisions、evidence、compact boundaries

5. “本地参考层”重新定义为本地参考源

Passport Store 不是用来“降服”LLM 的，而是用来降低 LLM 漂移后的损害范围。

更准确的技术表述应该是：

  • LLM 是 candidate generator，不是参考源
  • Passport Store 是 local reference store，不是万能控制器
  • 错误输出要被回复校验器拦住，不能直接污染本地参考层
  • 高风险动作不能只靠自然语言理解，必须过策略和确认

五、当前已经真正落地的部分

  • 产品定位、MVP、安全架构已经重写，并能通过 API 直接读取
  • 设备运行态已经支持风险分级策略，不再把多签当所有动作的默认机制
  • runtime search 已经明确返回 local_first_non_vector、lexical_v1 和 vectorUsed=false
  • 删除 / 撤销类 critical 动作会在 runtime 中被判成 multisig 冷路径
  • 本地纪要、decision、evidence、compact boundary 已经进入恢复链

六、哪些还没有真正解决

下面这些问题现在不是“已解决”，而是“方向已经对、代码还要继续补”。
  • 本地存储加密：目前仍需作为下一阶段 P0 真正实现
  • 受限执行层：目前风险分级已有，真正的隔离执行环境仍未完成
  • 法律责任条款：需要正式的产品协议和企业部署条款支持
  • 平台生态适配：当前只解决了战略收缩，没有拿到大厂平台入口
  • 成本模型：当前是方向正确，还需要补量化测算与部署方案

七、能否落地

能落地，但前提是继续按当前收缩后的路线走，而不是再回到“全球协议”的大叙事。

现实可落地路径应当是：

  1. 先把 single-device resident agent runtime 做稳
  2. 把本地加密和受限执行层补成 P0
  3. 把 connector 定位成“授权接入”，而不是“全网接管”
  4. 把高风险动作、恢复能力、审计能力做成真正的卖点

八、最终判断

最终判断：Agent Passport 作为“全球 Agent 协议”现在不成立；作为“单机单 Agent、本地优先、风险分级、关键动作可校验的 Agent Runtime”是成立的，而且已经进入可继续打磨的内部 Alpha 阶段。
