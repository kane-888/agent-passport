# agent-passport Operator 安全手册

## 目标

这份手册只回答 3 个问题：

1. 现在是谁该做决定
2. 当前姿态下先做什么
3. 什么情况下才能继续写入、执行或切回正常运行

不要把它当产品介绍。它是值班手册。

## 一页决策

先按这 4 条判断，不要跳步：

1. `securityPosture.mode != normal`：先锁边界，不讨论放行。
2. `formalRecoveryFlow.durableRestoreReady != true`：先补正式恢复主线，不把自动恢复当完成。
3. `constrainedExecution.status in degraded|locked`：先停真实执行，只保留读取、恢复和证据保全。
4. 只有前三条都通过，再看 `crossDeviceRecoveryClosure.readyForRehearsal` / `readyForCutover` 决定能不能演练或切机。

只有下面 4 条同时成立，才允许恢复正常或允许真实切机：

- 当前姿态回到 `normal`
- `formalRecoveryFlow.durableRestoreReady=true`
- 受限执行层不在 `degraded` / `locked`
- 目标机器核验通过，且 `readyForCutover=true`

## 角色责任

| 角色 | 负责什么 | 不负责什么 |
| --- | --- | --- |
| 持有者 / 委托主体 | 决定是否继续业务、是否接受恢复结果、是否允许恢复后重新放开写入/执行 | 不负责临时改代码、绕过门禁 |
| 运营者 / 值班操作员 | 切安全姿态、保全现场、导出证据、执行恢复包/演练/初始化包流程、记录操作结果 | 不负责替持有者做业务判断 |
| 平台 / 开发维护 | 定位根因、修复缺陷、提供回放与迁移工具、解释系统状态 | 不负责跳过持有者批准直接恢复业务 |

## 先看什么

出现异常时，先按这个顺序看：

1. `/` 的“公开健康度”
2. `/operator`
3. `/` 的“正式恢复周期”
4. `/` 的“自动恢复边界”
5. `/api/security`
6. `/api/device/setup`
7. 需要做清理判断时，再看 `/lab.html` 的维护减旧面板

如果这 7 个位置给出的结论不一致，先按更保守的结论执行。
只有需要受保护修复证据时再进 `/repair-hub`；只有需要线程成员 / 启动真值时再进 `/offline-chat`。不要把 `/` 当成万能入口，也不要把 `/repair-hub` 当成值班判断面。

## 姿态动作表

| 姿态 | 什么时候用 | 当前目标 | 运营者立刻做什么 | 持有者需要拍板什么 | 退出条件 |
| --- | --- | --- | --- | --- | --- |
| `normal` | 无异常，或异常已经收束 | 保持基线新鲜，不带病运行 | 确认最近恢复演练仍在窗口内；确认受限执行没有退化；确认自动恢复没有 `loop_detected / failed` | 是否允许继续正常运行 | 正常持续 |
| `read_only` | 怀疑错误写入、状态污染、账本需要保全 | 先停写，保住现场 | 停止新增写入；导出 `/api/security` 与 `/api/device/setup` 摘要；记录最新 recovery / rehearsal / setup package 证据 | 是否继续只读排查，还是升级到 `disable_exec` / `panic` | 已确认没有继续污染风险，且恢复/排查完成 |
| `disable_exec` | 怀疑执行链有问题，但读和恢复仍要继续 | 停执行，保留读和恢复能力 | 停止受限执行、runner 执行动作和外部操作；保留读取、恢复包导出、恢复演练；检查受限执行退化点 | 是否允许恢复后重新放开执行 | 执行面风险已解释清楚，受限执行边界恢复正常 |
| `panic` | 怀疑密钥泄露、越权执行扩大、异常正在扩散 | 先锁住写入、执行和外网，保全证据 | 立即保全 `/api/security`、runner history、受限执行审计、恢复证据；只保留安全维护入口；准备密钥轮换或切机恢复 | 是否进入灾备切换、是否接受业务中断 | 已完成证据保全、风险面收敛、恢复目标明确 |

## 前 15 分钟动作

### 发现异常后的第 0 到 5 分钟

- 先判断是否需要切到 `read_only`、`disable_exec` 或 `panic`
- 记录异常时间、触发动作、当前页面/线程、最近一次 runner/audit
- 导出 `/api/security` 和 `/api/device/setup` 当前结果

### 第 5 到 10 分钟

- 确认最近一次恢复演练是否仍在策略窗口内
- 确认最近恢复包与初始化包是否可用
- 确认当前问题属于写入污染、执行风险、还是恢复缺口

### 第 10 到 15 分钟

- 如果是写入污染：保持 `read_only`，先做证据保全和恢复判断
- 如果是执行风险：切 `disable_exec` 或更高姿态，停止继续操作宿主系统
- 如果是密钥/外部扩散风险：直接 `panic`

## 决策规则

### 什么时候必须升级姿态

- 发现未经确认的高风险执行
- 怀疑密钥泄露或 token 泄露
- 自动恢复进入循环且继续重试可能放大损害
- 受限执行层退化后仍出现真实执行需求

### 什么时候不能直接恢复正常

- 最近恢复演练过期
- 当前正式恢复流程仍有必填缺口
- 受限执行层处于退化状态
- 根因还没解释清楚，只是“看起来恢复了”

## 证据清单

每次异常至少保留这些东西：

- `/api/security`
- `/api/device/setup`
- 最近一次自动恢复闭环审计
- 最近一次受限执行审计
- 最近一次 recovery rehearsal 结果
- 当前安全姿态切换前后的时间点

如果问题涉及执行或越权，再额外保留：

- 请求 capability
- gate reasons
- broker / worker / system sandbox 状态
- 是否有 fallback activated

## 恢复与交接

如果异常进入恢复阶段，不要直接口头交接。交接内容最少包括：

- 当前安全姿态
- 当前正式恢复下一步
- 最近一次通过的恢复演练时间
- 最近恢复包和初始化包的时间与摘要
- 现在阻止恢复正常的唯一原因

## 一句话收口

异常时不要先问“能不能继续跑”，先问“谁来决策、先锁哪里、证据保住没有”。
